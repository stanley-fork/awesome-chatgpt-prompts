A engenharia de contexto decide **o que o modelo pode ver**. A engenharia de loop decide **o que o sistema fará a seguir**.

Em vez de uma pessoa ler repetidamente uma resposta e escrever a próxima solicitação, um loop transforma esse trabalho de acompanhamento em um sistema projetado. Ele dá um objetivo à IA, deixa-a agir, observa o feedback real, avalia o resultado e se adapta ou para.

<Callout type="info" title="A breve definição">
A engenharia de loop é a prática de projetar o ciclo de feedback em torno de um sistema de IA: o objetivo, as ações, as observações, a avaliação, o estado, as proteções e as regras de parada que movem o trabalho em direção a um resultado verificável.
</Callout>

## De um bom prompt a um bom loop

Um prompt pode produzir uma primeira tentativa forte. Um loop é útil quando o número de etapas não pode ser conhecido antecipadamente, o ambiente pode mudar ou a primeira tentativa deve ser verificada em relação a evidências.

Esta ideia baseia-se em padrões de agentes anteriores. O [artigo ReAct](https://arxiv.org/abs/2210.03629) mostrou o valor de intercalar ações com observações de um ambiente externo. O [padrão avaliador-otimizador] (https://www.anthropic.com/engineering/building-effective-agents) da Anthropic adiciona uma etapa de feedback distinta, enquanto o [guia prático para agentes] (https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) da OpenAI descreve as execuções do agente como loops governados por condições de saída. O termo mais recente **engenharia de loop** concentra a atenção no projeto deliberado de todo esse ciclo.

## Escreva o contrato de loop primeiro

Antes de escolher um modelo ou estrutura, redija um pequeno contrato. Se esses campos forem vagos, o loop transformará a ambiguidade em custo repetido.

```yaml
goal: "Qual estado observável deve se tornar verdadeiro?"
inputs: "O que inicia o loop?"
state: "Que fatos e tentativas persistem entre as iterações?"
actions: "Quais ferramentas o sistema pode usar e com quais permissões?"
observations: "Que sinais externos retornam dessas ações?"
evaluator: "Quais rubricas ou verificações determinísticas avaliam o resultado?"
success: "Que evidências provam que o objetivo está completo?"
failure: "Quais condições requerem recuperação ou ajuda humana?"
budget: "Máximo de turnos, tempo, fichas, dinheiro ou efeitos colaterais"
```

<Callout type="warning" title="Um loop sem saída é um modo de falha">
Sempre defina três saídas: **sucesso** quando a evidência satisfaz a meta, **fracasso** quando a recuperação não é mais útil e **orçamento esgotado** quando o ciclo atinge seu custo permitido ou limite de risco.
</Callout>

## As observações devem vir do mundo

O modelo que diz “isso parece correto” não é uma evidência forte. Uma observação útil é produzida por algo fora da resposta que está sendo julgada.

## Separe o criador do verificador

Para trabalhos de baixo risco, um modelo pode ser elaborado e autoavaliado. Para trabalhos importantes, use um verificador com função diferente e autoridade limitada.

<InfoGrid columns={2} items={[
  { label: "Fabricante", description: "Propõe a mudança, convoca ferramentas de ação e explica quais evidências devem ser coletadas.", color: "amber" },
  { label: "Verificador", description: "Recebe o objetivo, o artefato e as evidências; aplica a rubrica; não pode reescrever silenciosamente o trabalho que avalia.", color: "green" },
]} />

O verificador não precisa ser outra IA. Prefira o avaliador mais determinístico disponível:

1. **Verificações exatas** — tipos, esquemas, restrições, permissões, hashes
2. **Verificações executáveis** — testes, linters, simulações, validação de link
3. **Verificações de rubricas** — um modelo independente ou humano usando critérios nomeados
4. **Verificações de resultados** — comportamento real do usuário ou métricas de produção ao longo do tempo

<Callout type="tip" title="Confiança não é evidência">
Uma pontuação de confiança gerada pelo mesmo modelo que criou o artefato ainda é uma saída do modelo. Trate isso como uma dica de roteamento, não como uma prova.
</Callout>

## O estado é a espinha dorsal do loop

Contexto é o que o modelo vê **neste turno**. Estado é o registro compacto que permite que a próxima curva continue sem repetir ou esquecer o trabalho.

<Checklist title="Estado de loop útil" items={[
  { text: "A meta atual e os critérios de aceitação" },
  { text: "Ações já tentadas e seus resultados observáveis" },
  { text: "Artefatos produzidos, com versões ou referências estáveis" },
  { text: "Veredictos do avaliador e questões não resolvidas" },
  { text: "Orçamento consumido e permissões ainda disponíveis" },
  { text: "O motivo para continuar, parar ou escalar" },
]} />

Armazene decisões e evidências concisas – não a cadeia de pensamento privada de um modelo. O bom estado é pequeno, inspecionável e seguro para ser retomado.

## Formas de loop comuns

### O ciclo de reparo

`reproduzir → alterar uma coisa → executar verificações → diagnosticar → repetir ou parar`

Melhor para código, configuração, limpeza de dados e qualquer tarefa com feedback executável.

### O ciclo avaliador-otimizador

`gerar → pontuar com uma rubrica → retornar feedback direcionado → revisar`

Melhor para redação, tradução, crítica de design e resultados cuja qualidade melhora por meio de feedback articulado.

### O ciclo de pesquisa

`pesquisar → inspecionar fontes → identificar lacunas ou conflitos → pesquisar novamente → sintetizar`

Melhor quando a integralidade não é conhecida antecipadamente. A regra de parada deve medir a cobertura das evidências, não o número de resultados da pesquisa.

### O Loop Humano

`preparar → verificar automaticamente → pausar antes de ação de alto risco → aprovação ou redirecionamento humano`

Melhor para pagamentos, publicação, exclusão, alterações de acesso, decisões médicas ou legais e outras ações consequentes.

## Uma implementação mínima

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

O código é a parte fácil. As questões difíceis são questões de domínio: Qual teste prova que o bug desapareceu? Qual fonte é oficial? Qual ação é reversível? Quem pode aprovar a etapa final? Essas decisões são o verdadeiro trabalho da engenharia de loop.

## Resumo

<Callout type="tip" title="Crie o feedback, não apenas o prompt">
Um ciclo confiável tem um objetivo mensurável, ações limitadas, observações confiáveis, um avaliador explícito, estado compacto, saídas seguras e julgamento humano onde as consequências assim o exigirem. O modelo está dentro do loop; o engenheiro permanece responsável pelo loop.
</Callout>
