L'ingegneria del contesto decide **cosa può vedere il modello**. L'ingegneria del loop decide **cosa farà il sistema dopo**.

Invece di una persona che legge ripetutamente una risposta e scrive il messaggio successivo, un ciclo trasforma il lavoro di follow-up in un sistema progettato. Assegna un obiettivo all’intelligenza artificiale, la lascia agire, osserva il feedback reale, valuta il risultato e si adatta o si ferma.

<Callout type="info" title="La breve definizione">
L'ingegneria del loop è la pratica di progettare il ciclo di feedback attorno a un sistema di intelligenza artificiale: obiettivo, azioni, osservazioni, valutazione, stato, guardrail e regole di arresto che spostano il lavoro verso un risultato verificabile.
</Callout>

## Da un buon suggerimento a un buon loop

Un prompt può produrre un primo tentativo efficace. Un ciclo è utile quando non è possibile conoscere in anticipo il numero di passaggi, l'ambiente può cambiare o il primo tentativo deve essere confrontato con prove.

Questa idea si basa su modelli di agenti precedenti. Il [documento ReAct](https://arxiv.org/abs/2210.03629) ha mostrato il valore dell'interlacciamento delle azioni con osservazioni provenienti da un ambiente esterno. Il [modello valutatore-ottimizzatore](https://www.anthropic.com/engineering/building-effective-agents) di Anthropic aggiunge un passaggio di feedback distinto, mentre la [guida pratica agli agenti](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) di OpenAI descrive le esecuzioni degli agenti come cicli governati da condizioni di uscita. Il termine più recente **ingegneria del circuito** concentra l'attenzione sulla progettazione deliberata dell'intero ciclo.

## Scrivi prima il contratto di loop

Prima di scegliere un modello o una struttura, scrivi un piccolo contratto. Se questi campi sono vaghi, il ciclo trasformerà l’ambiguità in costi ripetuti.

```yaml
goal: "Quale stato osservabile dovrebbe diventare vero?"
inputs: "Cosa dà inizio al ciclo?"
state: "Quali fatti e tentativi persistono tra le iterazioni?"
actions: "Quali strumenti può utilizzare il sistema e con quali autorizzazioni?"
observations: "Quali segnali esterni ritornano da quelle azioni?"
evaluator: "Quali rubriche o controlli deterministici giudicano il risultato?"
success: "Quali prove dimostrano che l’obiettivo è stato raggiunto?"
failure: "Quali condizioni richiedono il recupero o l’aiuto umano?"
budget: "Turni massimi, tempo, gettoni, denaro o effetti collaterali"
```

<Callout type="warning" title="Un ciclo senza uscita è una modalità di fallimento">
Definire sempre tre uscite: **successo** quando le prove soddisfano l'obiettivo, **fallimento** quando il recupero non è più utile e **budget esaurito** quando il ciclo raggiunge il limite di costo o rischio consentito.
</Callout>

## Separare il creatore dalla pedina

Per i lavori a basso rischio, un modello può essere redatto e sottoposto a revisione autonoma. Per lavori importanti, utilizzare una pedina con un lavoro diverso e autorità limitata.

<InfoGrid columns={2} items={[
  { label: "Creatore", description: "Propone il cambiamento, invoca strumenti di azione e spiega quali prove dovrebbero essere raccolte.", color: "amber" },
  { label: "Controllore", description: "Riceve l'obiettivo, l'artefatto e le prove; applica la rubrica; non può riscrivere silenziosamente il lavoro che valuta.", color: "green" },
]} />

La pedina non deve necessariamente essere un'altra IA. Preferisci il valutatore più deterministico disponibile:

1. **Controlli esatti**: tipi, schemi, vincoli, autorizzazioni, hash
2. **Controlli eseguibili**: test, linter, simulazioni, convalida dei collegamenti
3. **Controlli delle rubriche**: un modello indipendente o un essere umano che utilizza criteri denominati
4. **Controlli dei risultati**: comportamento reale degli utenti o parametri di produzione nel tempo

<Callout type="tip" title="La fiducia non è una prova">
Un punteggio di confidenza generato dallo stesso modello che ha creato l'artefatto è ancora un output del modello. Trattalo come un suggerimento di routing, non come una prova.
</Callout>

## Lo Stato è la spina dorsale del ciclo

Il contesto è ciò che il modello vede **in questo turno**. Lo Stato è il registro compatto che consente di continuare la svolta successiva senza ripetere o dimenticare il lavoro.

<Checklist title="Stato del ciclo utile" items={[
  { text: "L’obiettivo attuale e i criteri di accettazione" },
  { text: "Azioni già tentate e relativi risultati osservabili" },
  { text: "Artefatti prodotti, con versioni o riferimenti stabili" },
  { text: "Verdetti dei valutatori e questioni irrisolte" },
  { text: "Budget consumato e autorizzazioni ancora disponibili" },
  { text: "Il motivo per continuare, interrompere o intensificare" },
]} />

Archivia decisioni e prove concise, non la catena di pensiero privata di un modello. Il buono stato è piccolo, ispezionabile e sicuro da riprendere.

## Forme di loop comuni

### Il ciclo di riparazione

`riproduci → cambia una cosa → esegui controlli → diagnostica → ripeti o interrompi

Ideale per codice, configurazione, pulizia dei dati e qualsiasi attività con feedback eseguibile.

### Il ciclo valutatore-ottimizzatore

`genera → valuta con una rubrica → restituisce un feedback mirato → rivede`

Ideale per la scrittura, la traduzione, la critica progettuale e i risultati la cui qualità migliora attraverso un feedback articolato.

### Il ciclo della ricerca

`cerca → esamina le fonti → identifica lacune o conflitti → cerca di nuovo → sintetizza`

Meglio quando la completezza non è nota in anticipo. La regola di stop dovrebbe misurare la copertura delle prove, non il numero di risultati della ricerca.

### Il circuito controllato dall’uomo

"preparare → verificare automaticamente → mettere in pausa prima di un'azione ad alto rischio → l'utente approva o reindirizza".

Ideale per pagamenti, pubblicazione, cancellazione, modifiche di accesso, decisioni mediche o legali e altre azioni consequenziali.

## Un'implementazione minima

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

Il codice è la parte facile. Le domande difficili sono domande di dominio: quale test dimostra che il bug è scomparso? Quale fonte è autorevole? Quale azione è reversibile? Chi può approvare il passaggio finale? Tali decisioni sono il vero lavoro dell’ingegneria del loop.
