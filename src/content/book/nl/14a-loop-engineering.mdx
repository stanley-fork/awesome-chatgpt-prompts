Context-engineering bepaalt **wat het model kan zien**. Loop-engineering beslist **wat het systeem vervolgens doet**.

In plaats van dat iemand herhaaldelijk een antwoord leest en de volgende prompt schrijft, verandert een lus dat vervolgwerk in een ontworpen systeem. Het geeft een AI een doel, laat hem handelen, observeert echte feedback, evalueert het resultaat en past zich aan of stopt.

<Callout type="info" title="De korte definitie">
Loop engineering is de praktijk van het ontwerpen van de feedbackcyclus rond een AI-systeem: het doel, de acties, observaties, evaluatie, status, vangrails en stopregels die het werk in de richting van een verifieerbaar resultaat brengen.
</Callout>

## Van een goede prompt naar een goede loop

Een prompt kan een sterke eerste poging opleveren. Een lus is handig wanneer het aantal stappen vooraf niet bekend is, de omgeving kan veranderen of de eerste poging moet worden getoetst aan bewijsmateriaal.

Dit idee bouwt voort op eerdere agentpatronen. Het [ReAct-paper](https://arxiv.org/abs/2210.03629) toonde de waarde aan van het combineren van acties met observaties uit een externe omgeving. Anthropic's [evaluator-optimizer-patroon](https://www.anthropic.com/engineering/building-effective-agents) voegt een duidelijke feedbackstap toe, terwijl OpenAI's [praktische gids voor agenten](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) agentruns beschrijft als lussen die worden beheerst door exitvoorwaarden. De nieuwere term **loop engineering** vestigt de aandacht op het doelbewust ontwerpen van die hele cyclus.

## Schrijf eerst het luscontract

Voordat u een model of raamwerk kiest, schrijft u een klein contract. Als deze velden vaag zijn, zal de lus dubbelzinnigheid omzetten in herhaalde kosten.

```yaml
goal: "Welke waarneembare toestand zou waar moeten worden?"
inputs: "Wat begint de lus?"
state: "Welke feiten en pogingen blijven bestaan ​​tussen de iteraties?"
actions: "Welke tools mag het systeem gebruiken en met welke rechten?"
observations: "Welke externe signalen komen uit die acties terug?"
evaluator: "Welke rubriek- of deterministische controles beoordelen het resultaat?"
success: "Welk bewijs bewijst dat het doel voltooid is?"
failure: "Welke aandoeningen vereisen herstel of menselijke hulp?"
budget: "Maximale beurten, tijd, tokens, geld of bijwerkingen"
```

<Callout type="warning" title="Een lus zonder uitgang is een foutmodus">
Definieer altijd drie uitgangen: **succes** wanneer bewijsmateriaal voldoet aan het doel, **mislukking** wanneer herstel niet langer nuttig is, en **budget uitgeput** wanneer de lus de toegestane kosten- of risicogrens bereikt.
</Callout>

## Scheid de Maker van de Checker

Voor werk met een laag risico kan één model worden opgesteld en zelf worden beoordeeld. Gebruik voor belangrijk werk een controleur met een andere taak en beperkte bevoegdheid.

<InfoGrid columns={2} items={[
  { label: "Maker", description: "Stelt de verandering voor, roept actie-instrumenten op en legt uit welk bewijsmateriaal moet worden verzameld.", color: "amber" },
  { label: "Controleur", description: "Ontvangt het doel, het artefact en het bewijsmateriaal; past de rubriek toe; kan het werk dat het beoordeelt niet stilletjes herschrijven.", color: "green" },
]} />

De controleur hoeft geen andere AI te zijn. Geef de voorkeur aan de meest deterministische beoordelaar die beschikbaar is:

1. **Exacte controles** — typen, schema's, beperkingen, machtigingen, hashes
2. **Uitvoerbare controles** — tests, linters, simulaties, linkvalidatie
3. **Rubriekcontroles**: een onafhankelijk model of mens die benoemde criteria gebruikt
4. **Uitkomstcontroles**: echt gebruikersgedrag of productiestatistieken in de loop van de tijd

<Callout type="tip" title="Vertrouwen is geen bewijs">
Een betrouwbaarheidsscore die wordt gegenereerd door hetzelfde model dat het artefact heeft gemaakt, is nog steeds een modeluitvoer. Beschouw het als een routehint, niet als bewijs.
</Callout>

## Staat is de ruggengraat van de lus

Context is wat het model **deze beurt** ziet. State is het compacte record waarmee de volgende beurt kan doorgaan zonder werk te herhalen of te vergeten.

<Checklist title="Nuttige lusstatus" items={[

Bewaar beknopte beslissingen en bewijsmateriaal, niet de persoonlijke gedachtegang van een model. Een goede staat is klein, controleerbaar en veilig om te hervatten.

## Gemeenschappelijke lusvormen

### De reparatielus

`reproduceer → verander één ding → voer controles uit → diagnosticeer → herhaal of stop`

Beste voor code, configuratie, gegevensopschoning en elke taak met uitvoerbare feedback.

### De Evaluator-Optimizer-loop

`genereren → scoren met een rubriek → gerichte feedback geven → herzien`

Het beste voor schrijven, vertalen, ontwerpkritiek en output waarvan de kwaliteit verbetert door gearticuleerde feedback.

### De onderzoekslus

`zoeken → bronnen inspecteren → hiaten of conflicten identificeren → opnieuw zoeken → synthetiseren`

Het beste als de volledigheid niet vooraf bekend is. De stopregel moet de bewijsdekking meten, en niet het aantal zoekresultaten.

### De mensgerichte lus

`voorbereiden → automatisch verifiëren → pauzeren vóór actie met een hoog risico → menselijke goedkeuring of omleiding`

Het beste voor betalingen, publicatie, verwijdering, toegangswijzigingen, medische of juridische beslissingen en andere daaruit voortvloeiende acties.

## Een minimale implementatie

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

De code is het makkelijke gedeelte. De moeilijke vragen zijn domeinvragen: welke test bewijst dat de bug verdwenen is? Welke bron is gezaghebbend? Welke actie is omkeerbaar? Wie kan de laatste stap goedkeuren? Deze beslissingen zijn het eigenlijke werk van loop engineering.

</Callout>
