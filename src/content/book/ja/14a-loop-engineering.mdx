コンテキスト エンジニアリングは、**モデルが認識できる内容**を決定します。ループ エンジニアリングは **システムが次に何を行うか** を決定します。

人間が繰り返し答えを読んで次のプロンプトを書くのではなく、ループがそのフォローアップ作業を設計されたシステムに変えます。 AI に目標を与え、行動させ、実際のフィードバックを観察し、結果を評価し、適応または停止します。

<Callout type="info" title="簡単な定義">
ループ エンジニアリングは、AI システムに関するフィードバック サイクル (目標、アクション、観察、評価、状態、ガードレール、検証可能な結果に向けて作業を進める停止ルール) を設計する実践です。
</Callout>

## 良いプロンプトから良いループへ

プロンプトにより、最初の試行が強力になる可能性があります。ループは、ステップ数が事前にわからない場合、環境が変化する可能性がある場合、または最初の試行を証拠と照合する必要がある場合に役立ちます。

## 最初にループ コントラクトを作成する

モデルまたはフレームワークを選択する前に、小さな契約を作成します。これらのフィールドがあいまいな場合、ループによってあいまいさが反復コストに変わります。

```yaml
goal: "どのような観察可能な状態が真になるべきでしょうか?"
inputs: "何がループを開始しますか?"
state: "どのような事実と試みが反復間で持続するか?"
actions: "システムはどのツールをどのような権限で使用できますか?"
observations: "それらのアクションからどのような外部信号が返されますか?"
evaluator: "どのルーブリックまたは決定論的なチェックが結果を判断しますか?"
success: "目標が完了したことを証明する証拠は何ですか?"
failure: "回復や人間の助けが必要な状態はどれですか?"
budget: "最大ターン、時間、トークン、お金、または副作用"
```

<Callout type="warning" title="出口のないループは障害モードです">
常に 3 つの出口を定義します。証拠が目標を満たした場合は **成功**、回復が役に立たなくなった場合は **失敗**、ループが許容コストまたはリスクの境界に達した場合は **予算が使い果たされた**。
</Callout>

「これは正しいようだ」というモデルは強力な証拠ではありません。有用な観察は、答えの外側にある何かが判断されることによって生み出されます。

## メーカーとチェッカーを分離する

リスクの低い作業の場合、1 つのモデルで草案を作成して自己レビューできます。重要な作業については、別のジョブと制限された権限を持つチェッカーを使用してください。

<InfoGrid columns={2} items={[
  { label: "メーカー", description: "変更を提案し、アクションツールを呼び出し、どのような証拠を収集する必要があるかを説明します。", color: "amber" },
  { label: "チェッカー", description: "目標、成果物、証拠を受け取ります。ルーブリックを適用します。採点した作品を黙って書き直すことはできません。", color: "green" },
]} />

チェッカーは別の AI である必要はありません。利用可能な最も決定的な評価器を優先します。

1. **正確なチェック** — 型、スキーマ、制約、権限、ハッシュ
2. **実行可能チェック** — テスト、リンター、シミュレーション、リンク検証
3. **ルーブリック チェック** — 名前付き基準を使用する独立したモデルまたは人間
4. **結果チェック** — 実際のユーザーの行動または長期にわたる運用指標

<Callout type="tip" title="自信は証拠ではない">
アーティファクトを作成したのと同じモデルによって生成された信頼スコアは、依然としてモデルの出力です。これを証拠ではなくルーティングのヒントとして扱います。
</Callout>

## 状態はループの背骨である

コンテキストは、**このターン**でモデルが見ているものです。状態は、作業を繰り返したり忘れたりすることなく次のターンを継続できるコンパクトな記録です。

<Checklist title="有用なループ状態" items={[

  { text: "すでに試行されたアクションとその観察可能な結果" },

モデルのプライベートな思考連鎖ではなく、簡潔な決定と証拠を保管してください。良好な状態は小さく、検査可能であり、安全に再開できます。

## 一般的なループ形状

### 修理ループ

「再現→何かを変更→チェックを実行→診断→繰り返すか停止」

コード、構成、データのクリーンアップ、および実行可能なフィードバックを伴うタスクに最適です。

### 評価者と最適化者のループ

「生成 → ルーブリックでスコア付け → 的を絞ったフィードバックを返す → 修正」

執筆、翻訳、デザイン批評、明確なフィードバックによって品質が向上する成果物に最適です。

### 研究ループ

`検索 → ソースを検査 → ギャップや矛盾を特定 → 再検索 → 合成`

完全性が事前にわからない場合に最適です。停止ルールは、検索結果の数ではなく、証拠の範囲を測定する必要があります。

### ヒューマン・ゲート・ループ

「準備→自動的に検証→リスクの高いアクションの前に一時停止→人間が承認またはリダイレクト」

支払い、公開、削除、アクセス変更、医学的または法的決定、その他の結果的なアクションに最適です。

## 最小限の実装

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

コードは簡単な部分です。難しい質問はドメインに関する質問です。バグがなくなったことを証明するのはどのテストですか?どの情報源が信頼できるのでしょうか?どのアクションが元に戻せるのでしょうか?最終ステップを承認できるのは誰ですか?これらの決定はループ エンジニアリングの実際の作業です。
