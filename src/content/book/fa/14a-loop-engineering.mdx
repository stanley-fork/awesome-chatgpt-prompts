مهندسی زمینه تصمیم می‌گیرد **چه چیزی را مدل می‌تواند ببیند**. مهندسی حلقه تصمیم می‌گیرد **سیستم بعدی چه کاری انجام دهد**.

به جای اینکه فردی به طور مکرر پاسخی را بخواند و دستور بعدی را بنویسد، یک حلقه آن کار پیگیری را به یک سیستم طراحی شده تبدیل می کند. به هوش مصنوعی یک هدف می دهد، به آن اجازه می دهد عمل کند، بازخورد واقعی را مشاهده می کند، نتیجه را ارزیابی می کند، و یا سازگار می شود یا متوقف می شود.

<Callout type="info" title="تعریف کوتاه">
مهندسی حلقه تمرین طراحی چرخه بازخورد حول یک سیستم هوش مصنوعی است: هدف، اقدامات، مشاهدات، ارزیابی، وضعیت، نرده‌های محافظ و قوانین توقف که کار را به سمت یک نتیجه قابل تأیید پیش می‌برد.
</Callout>

## از یک اعلان خوب به یک حلقه خوب

یک درخواست می تواند اولین تلاش قوی را ایجاد کند. یک حلقه زمانی مفید است که تعداد مراحل را نمی توان از قبل مشخص کرد، محیط می تواند تغییر کند یا اولین تلاش باید با شواهد بررسی شود.

## ابتدا قرارداد حلقه را بنویسید

قبل از انتخاب مدل یا چارچوب، یک قرارداد کوچک بنویسید. اگر این فیلدها مبهم باشند، حلقه ابهام را به هزینه مکرر تبدیل می کند.

```yaml
goal: "کدام حالت قابل مشاهده باید درست شود؟"
inputs: "چه چیزی حلقه را شروع می کند؟"
state: "چه حقایق و تلاش هایی در بین تکرارها ادامه دارند؟"
actions: "سیستم از چه ابزارهایی و با چه مجوزهایی می تواند استفاده کند؟"
observations: "چه سیگنال های خارجی از آن اقدامات بازمی گردد؟"
evaluator: "کدام دسته بندی یا چک های قطعی نتیجه را قضاوت می کنند؟"
success: "چه شواهدی ثابت می کند هدف کامل است؟"
failure: "کدام شرایط نیاز به بهبودی یا کمک انسانی دارد؟"
budget: "حداکثر نوبت، زمان، نشانه، پول یا عوارض جانبی"
```

<Callout type="warning" title="یک حلقه بدون خروجی یک حالت شکست است">
همیشه سه خروجی تعریف کنید: **موفقیت** زمانی که شواهد هدف را برآورده می کند، **شکست** زمانی که بازیابی دیگر مفید نیست، و **بودجه تمام می شود** زمانی که حلقه به مرز هزینه یا ریسک مجاز خود می رسد.
</Callout>

]} />

لازم نیست جستجوگر یک هوش مصنوعی دیگر باشد. قطعی ترین ارزیاب موجود را ترجیح دهید:

1. **بررسی های دقیق** - انواع، طرحواره ها، محدودیت ها، مجوزها، هش ها
2. **بررسی های قابل اجرا** - تست ها، لینترها، شبیه سازی ها، اعتبارسنجی لینک
3. **بررسی روبریک** - یک مدل یا انسان مستقل با استفاده از معیارهای نامگذاری شده
4. **بررسی نتیجه** - رفتار واقعی کاربر یا معیارهای تولید در طول زمان

<Callout type="tip" title="اعتماد به نفس مدرک نیست">
یک امتیاز اطمینان ایجاد شده توسط همان مدلی که مصنوع را ساخته است، هنوز یک خروجی مدل است. آن را به عنوان یک اشاره مسیریابی تلقی کنید، نه اثبات.
</Callout>

## ایالت ستون فقرات حلقه است

زمینه چیزی است که مدل **این چرخش** را می بیند. State رکورد فشرده ای است که به دور بعدی اجازه می دهد بدون تکرار یا فراموش کردن کار ادامه یابد.

## شکل های حلقه مشترک

### حلقه تعمیر

«تکثیر → تغییر یک چیز → اجرای بررسی ها → تشخیص → تکرار یا توقف».

بهترین گزینه برای کد، پیکربندی، پاکسازی داده ها و هر کار با بازخورد قابل اجرا.

### حلقه ارزیاب-بهینه ساز

«ایجاد → امتیاز با یک عنوان → بازخورد هدفمند → بازبینی».

بهترین برای نوشتن، ترجمه، نقد طراحی، و خروجی هایی که کیفیت آنها از طریق بازخورد بیان شده بهبود می یابد.

### حلقه تحقیق

«جستجو → بازرسی منابع → شناسایی شکاف ها یا تضادها → جستجوی مجدد → ترکیب».

بهترین زمانی است که کامل بودن از قبل مشخص نباشد. قانون توقف باید پوشش شواهد را اندازه گیری کند، نه تعداد نتایج جستجو را.

### حلقه انسان-دردار

«آماده → تأیید خودکار → مکث قبل از اقدام پرخطر → تأیید یا تغییر مسیر توسط انسان».

بهترین گزینه برای پرداخت ها، انتشار، حذف، تغییرات دسترسی، تصمیمات پزشکی یا حقوقی و سایر اقدامات بعدی.

## اجرای حداقلی

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

کد بخش آسان است. سؤالات سخت سؤالات دامنه هستند: کدام آزمایش ثابت می کند که باگ برطرف شده است؟ کدام منبع معتبر است؟ کدام عمل برگشت پذیر است؟ چه کسی می تواند مرحله نهایی را تایید کند؟ این تصمیمات کار واقعی مهندسی حلقه هستند.
