La ingeniería de contexto decide **qué puede ver el modelo**. La ingeniería de bucle decide **qué hace el sistema a continuación**.

En lugar de que una persona lea repetidamente una respuesta y escriba la siguiente pregunta, un bucle convierte ese trabajo de seguimiento en un sistema diseñado. Le da un objetivo a la IA, le permite actuar, observa comentarios reales, evalúa el resultado y se adapta o se detiene.

<Callout type="info" title="La definición corta">
La ingeniería de bucles es la práctica de diseñar el ciclo de retroalimentación en torno a un sistema de IA: el objetivo, las acciones, las observaciones, la evaluación, el estado, las barreras de seguridad y las reglas de detención que mueven el trabajo hacia un resultado verificable.
</Callout>

## De un buen aviso a un buen bucle

Un aviso puede producir un primer intento sólido. Un bucle es útil cuando no se puede saber de antemano el número de pasos, el entorno puede cambiar o el primer intento debe compararse con la evidencia.

Esta idea se basa en patrones de agentes anteriores. El [artículo ReAct] (https://arxiv.org/abs/2210.03629) mostró el valor de entrelazar acciones con observaciones de un entorno externo. El [patrón de optimizador-evaluador](https://www.anthropic.com/engineering/building-effective-agents) de Anthropic agrega un paso de retroalimentación distinto, mientras que la [guía práctica para agentes](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) de OpenAI describe las ejecuciones de agentes como bucles gobernados por condiciones de salida. El término más nuevo **ingeniería de bucle** centra la atención en diseñar deliberadamente todo ese ciclo.

## Escriba el contrato de bucle primero

Antes de elegir un modelo o marco, escriba un pequeño contrato. Si estos campos son vagos, el bucle convertirá la ambigüedad en un costo repetido.

```yaml
goal: "¿Qué estado observable debería volverse verdadero?"
inputs: "¿Qué inicia el ciclo?"
state: "¿Qué hechos e intentos persisten entre iteraciones?"
actions: "¿Qué herramientas puede utilizar el sistema y con qué permisos?"
observations: "¿Qué señales externas provienen de esas acciones?"
evaluator: "¿Qué rúbrica o controles deterministas juzgan el resultado?"
success: "¿Qué evidencia prueba que la meta está completa?"
failure: "¿Qué condiciones requieren recuperación o ayuda humana?"
budget: "Máximo de turnos, tiempo, fichas, dinero o efectos secundarios"
```

<Callout type="warning" title="Un bucle sin salida es un modo de falla">
Defina siempre tres salidas: **éxito** cuando la evidencia satisface el objetivo, **fracaso** cuando la recuperación ya no es útil y **presupuesto agotado** cuando el ciclo alcanza su límite de costo o riesgo permitido.
</Callout>

## Las observaciones deben venir del mundo

El modelo que dice “esto parece correcto” no es una evidencia sólida. Una observación útil la produce algo externo a la respuesta que se está juzgando.

Por eso las herramientas son importantes: las pruebas, los navegadores, las bases de datos, los validadores y la revisión humana convierten una suposición interna en un resultado observable.

## Separe el creador del corrector

Para trabajos de bajo riesgo, un modelo puede redactarse y revisarse por sí mismo. Para trabajos importantes, utilice un inspector con un trabajo diferente y autoridad limitada.

<InfoGrid columns={2} items={[
  { label: "Fabricante", description: "Propone el cambio, pide herramientas de acción y explica qué evidencia se debe recopilar.", color: "amber" },
  { label: "Inspector", description: "Recibe el objetivo, el artefacto y la evidencia; aplica la rúbrica; no puede reescribir silenciosamente el trabajo que califica.", color: "green" },
]} />

El inspector no tiene por qué ser otra IA. Prefiera el evaluador más determinista disponible:

1. **Comprobaciones exactas**: tipos, esquemas, restricciones, permisos, hashes
2. **Comprobaciones ejecutables**: pruebas, linters, simulaciones, validación de enlaces
3. **Verificaciones de rúbricas**: un modelo o ser humano independiente que utiliza criterios nombrados
4. **Comprobaciones de resultados**: comportamiento real del usuario o métricas de producción a lo largo del tiempo

<Callout type="tip" title="La confianza no es evidencia">
Una puntuación de confianza generada por el mismo modelo que creó el artefacto sigue siendo un resultado del modelo. Trátelo como una pista de ruta, no como una prueba.
</Callout>

## El estado es la columna vertebral del circuito

El contexto es lo que ve el modelo **este turno**. Estado es el registro compacto que permite continuar el siguiente turno sin repetir ni olvidar el trabajo.

<Checklist title="Estado de bucle útil" items={[
  { text: "El objetivo actual y los criterios de aceptación." },
  { text: "Acciones ya intentadas y sus resultados observables." },
  { text: "Artefactos producidos, con versiones o referencias estables." },
  { text: "Veredictos del evaluador y cuestiones no resueltas" },
  { text: "Presupuesto consumido y permisos aún disponibles" },
  { text: "El motivo para continuar, detener o intensificar" },
]} />

Almacene evidencia y decisiones concisas, no la cadena de pensamiento privada de un modelo. El buen estado es pequeño, inspeccionable y seguro para reanudar.

## Formas de bucle comunes

### El circuito de reparación

`reproducir → cambiar una cosa → ejecutar comprobaciones → diagnosticar → repetir o detener`

Lo mejor para código, configuración, limpieza de datos y cualquier tarea con comentarios ejecutables.

### El ciclo evaluador-optimizador

`generar → calificar con una rúbrica → devolver comentarios específicos → revisar`

Lo mejor para redacción, traducción, crítica de diseño y resultados cuya calidad mejora a través de comentarios articulados.

### El circuito de investigación

`buscar → inspeccionar fuentes → identificar lagunas o conflictos → buscar de nuevo → sintetizar`

Es mejor cuando no se sabe de antemano si está completo. La regla de detención debe medir la cobertura de evidencia, no la cantidad de resultados de búsqueda.

### El circuito cerrado por humanos

`preparar → verificar automáticamente → pausar antes de una acción de alto riesgo → el ser humano aprueba o redirige`

Lo mejor para pagos, publicaciones, eliminaciones, cambios de acceso, decisiones médicas o legales y otras acciones consecuentes.

## Una implementación mínima

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

El código es la parte fácil. Las preguntas difíciles son preguntas de dominio: ¿Qué prueba demuestra que el error desapareció? ¿Qué fuente es autorizada? ¿Qué acción es reversible? ¿Quién puede aprobar el paso final? Esas decisiones son el trabajo real de la ingeniería de bucles.

## Resumen

<Callout type="tip" title="Genere comentarios, no solo mensajes">
Un bucle confiable tiene un objetivo mensurable, acciones limitadas, observaciones confiables, un evaluador explícito, estado compacto, salidas seguras y juicio humano cuando las consecuencias lo exigen. El modelo está dentro del bucle; el ingeniero sigue siendo responsable del bucle.
</Callout>
