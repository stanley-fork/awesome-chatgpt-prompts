Контекстная инженерия решает **что видит модель**. Проектирование циклов решает **что система будет делать дальше**.

Вместо того, чтобы человек постоянно читал ответ и писал следующую подсказку, цикл превращает последующую работу в продуманную систему. Он дает ИИ цель, позволяет ему действовать, наблюдает за реальной обратной связью, оценивает результат и либо адаптируется, либо останавливается.

<Callout type="info" title="Краткое определение">
Циклическая инженерия — это практика разработки цикла обратной связи вокруг системы ИИ: цель, действия, наблюдения, оценка, состояние, ограничения и правила остановки, которые приближают работу к проверяемому результату.
</Callout>

## От хорошей подсказки к хорошему циклу

Подсказка может привести к сильной первой попытке. Цикл полезен, когда количество шагов не может быть известно заранее, среда может измениться или первая попытка должна быть проверена на соответствие фактическим данным.

Новый термин «контурная инженерия» фокусирует внимание на сознательном проектировании всего цикла.

## Сначала напишите контракт цикла

Прежде чем выбрать модель или фреймворк, напишите небольшой договор. Если эти поля неопределенны, цикл превратит неоднозначность в повторяющиеся затраты.

```yaml
goal: "Какое наблюдаемое состояние должно стать истинным?"
inputs: "Что запускает цикл?"
state: "Какие факты и попытки сохраняются между итерациями?"
actions: "Какие инструменты может использовать система и с какими разрешениями?"
observations: "Какие внешние сигналы исходят от этих действий?"
evaluator: "По каким критериям или детерминированным проверкам оценивается результат?"
success: "Какие доказательства доказывают, что цель достигнута?"
failure: "Какие состояния требуют выздоровления или помощи человека?"
budget: "Максимальное количество ходов, время, жетоны, деньги или побочные эффекты"
```

<Callout type="warning" title="Цикл без выхода — это режим отказа">
Всегда определяйте три выхода: **успех**, когда данные соответствуют цели, **провал**, когда восстановление больше не является полезным, и **бюджет исчерпан**, когда цикл достигает допустимой границы затрат или риска.
</Callout>

## Отделите создателя от шашки

Для работы с низким уровнем риска одна модель может быть составлена ​​и подвергнута самопроверке. Для важной работы используйте проверяющего с другим заданием и ограниченными полномочиями.

<InfoGrid columns={2} items={[
  { label: "Создатель", description: "Предлагает изменения, называет инструменты действий и объясняет, какие доказательства следует собрать.", color: "amber" },
  { label: "Шашка", description: "Получает цель, артефакт и доказательства; применяет рубрику; не может молча переписать работу, которую оценивает.", color: "green" },
]} />

Шашкой не обязательно должен быть другой ИИ. Предпочитайте наиболее детерминированный оценщик:

1. **Точные проверки** — типы, схемы, ограничения, разрешения, хэши.
2. **Выполняемые проверки** — тесты, линтеры, моделирование, проверка ссылок.
3. **Проверка по критериям** — независимая модель или человек по указанным критериям.
4. **Проверка результатов** — реальное поведение пользователей или производственные показатели с течением времени.

<Callout type="tip" title="Уверенность – это не доказательство">
Оценка достоверности, полученная с помощью той же модели, которая создала артефакт, по-прежнему является выходными данными модели. Относитесь к этому как к подсказке маршрута, а не как к доказательству.
</Callout>

## Состояние — это позвоночник цикла

Контекст — это то, что модель видит **на этом повороте**. Состояние — это компактная запись, которая позволяет продолжить следующий ход, не повторяя и не забывая работу.

Храните краткие решения и доказательства, а не частную цепочку мыслей модели. Хорошее состояние — это небольшое состояние, которое можно проверить и его можно безопасно возобновить.

## Распространенные формы петель

### Ремонтный цикл

`воспроизвести → изменить что-то → запустить проверки → диагностировать → повторить или остановить`

Лучше всего подходит для кода, настройки, очистки данных и любых задач с исполняемой обратной связью.

### Цикл вычислителя-оптимизатора

`сгенерировать → оценить с помощью рубрики → вернуть целевую обратную связь → пересмотреть`

Лучше всего подходит для написания, перевода, критики дизайна и материалов, качество которых улучшается за счет четко сформулированной обратной связи.

### Цикл исследования

`поиск → проверка источников → выявление пробелов или противоречий → повторный поиск → синтезирование`

Лучше всего, когда полнота не известна заранее. Правило остановки должно измерять охват доказательств, а не количество результатов поиска.

### Петля, управляемая человеком

`подготовка → проверка автоматически → пауза перед действием высокого риска → человек одобряет или перенаправляет`

Лучше всего подходит для платежей, публикации, удаления, изменения доступа, медицинских или юридических решений и других последующих действий.

## Минимальная реализация

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

Код — это самая простая часть. Сложные вопросы — это вопросы предметной области: какой тест доказывает, что ошибка исчезла? Какой источник авторитетный? Какое действие обратимо? Кто может утвердить последний шаг? Эти решения представляют собой фактическую работу по проектированию контуров.
