Kontext-Engineering entscheidet, **was das Modell sehen kann**. Das Loop-Engineering entscheidet, **was das System als nächstes tut**.

Anstatt dass eine Person wiederholt eine Antwort liest und die nächste Eingabeaufforderung schreibt, verwandelt eine Schleife diese Folgearbeit in ein entworfenes System. Es gibt einer KI ein Ziel vor, lässt sie handeln, beobachtet echtes Feedback, wertet das Ergebnis aus und passt sich entweder an oder stoppt.

<Callout type="info" title="Die kurze Definition">
Beim Loop-Engineering handelt es sich um die Praxis, den Feedback-Zyklus rund um ein KI-System zu entwerfen: Ziel, Aktionen, Beobachtungen, Bewertung, Zustand, Leitplanken und Stoppregeln, die die Arbeit in Richtung eines überprüfbaren Ergebnisses bewegen.
</Callout>

## Von einer guten Eingabeaufforderung zu einer guten Schleife

Eine Aufforderung kann zu einem starken ersten Versuch führen. Eine Schleife ist nützlich, wenn die Anzahl der Schritte nicht im Voraus bekannt sein kann, sich die Umgebung ändern kann oder der erste Versuch anhand von Beweisen überprüft werden muss.

Diese Idee baut auf früheren Agentenmustern auf. Das [ReAct-Papier](https://arxiv.org/abs/2210.03629) zeigte den Wert der Verknüpfung von Aktionen mit Beobachtungen aus einer externen Umgebung. Das [Evaluator-Optimierer-Muster] (https://www.anthropic.com/engineering/building-effective-agents) von Anthropic fügt einen eindeutigen Feedback-Schritt hinzu, während der [praktische Leitfaden für Agenten] (https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) von OpenAI Agentenläufe als Schleifen beschreibt, die durch Exit-Bedingungen gesteuert werden. Der neuere Begriff „Loop Engineering“ konzentriert sich auf die bewusste Gestaltung des gesamten Zyklus.

## Schreiben Sie zuerst den Schleifenvertrag

Schreiben Sie einen kleinen Vertrag, bevor Sie ein Modell oder Framework auswählen. Wenn diese Felder vage sind, wird die Schleife die Mehrdeutigkeit in wiederholte Kosten umwandeln.

```yaml
goal: "Welcher beobachtbare Zustand sollte wahr werden?"
inputs: "Was startet die Schleife?"
state: "Welche Fakten und Versuche bleiben zwischen den Iterationen bestehen?"
actions: "Welche Tools darf das System nutzen und mit welchen Berechtigungen?"
observations: "Welche externen Signale gehen von diesen Aktionen aus?"
evaluator: "Welche Rubrik oder deterministische Prüfungen beurteilen das Ergebnis?"
success: "Welche Beweise beweisen, dass das Ziel erreicht ist?"
failure: "Welche Erkrankungen erfordern Genesung oder menschliche Hilfe?"
budget: "Maximale Anzahl an Runden, Zeit, Token, Geld oder Nebenwirkungen"
```

<Callout type="warning" title="Eine Schleife ohne Ausgang ist ein Fehlermodus">
Definieren Sie immer drei Ausgänge: **Erfolg**, wenn die Beweise das Ziel erfüllen, **Misserfolg**, wenn die Wiederherstellung nicht mehr nützlich ist, und **Budget erschöpft**, wenn die Schleife ihre zulässigen Kosten- oder Risikogrenzen erreicht.
</Callout>

## Beobachtungen müssen aus der Welt kommen

Die Aussage des Modells „Das sieht richtig aus“ ist kein überzeugender Beweis. Eine nützliche Beobachtung entsteht durch etwas außerhalb der zu beurteilenden Antwort.

## Trennen Sie den Maker vom Checker

Bei Arbeiten mit geringem Risiko kann ein Modell Entwürfe entwerfen und selbst überprüfen. Beauftragen Sie für wichtige Arbeiten einen Prüfer mit einer anderen Aufgabe und eingeschränkten Befugnissen.

<InfoGrid columns={2} items={[
  { label: "Hersteller", description: "Schlägt die Änderung vor, ruft Aktionsinstrumente auf und erklärt, welche Beweise gesammelt werden sollten.", color: "amber" },
  { label: "Checker", description: "Erhält das Ziel, das Artefakt und die Beweise; wendet die Rubrik an; kann die von ihm bewertete Arbeit nicht stillschweigend umschreiben.", color: "green" },
]} />

Der Prüfer muss keine andere KI sein. Bevorzugen Sie den deterministischsten verfügbaren Evaluator:

1. **Exakte Prüfungen** – Typen, Schemata, Einschränkungen, Berechtigungen, Hashes
2. **Ausführbare Prüfungen** – Tests, Linters, Simulationen, Linkvalidierung
3. **Rubrikprüfungen** – ein unabhängiges Modell oder ein unabhängiger Mensch anhand benannter Kriterien
4. **Ergebnisprüfungen** – echtes Benutzerverhalten oder Produktionsmetriken im Zeitverlauf

<Callout type="tip" title="Vertrauen ist kein Beweis">
Ein Konfidenzwert, der von demselben Modell generiert wurde, das das Artefakt erstellt hat, ist immer noch eine Modellausgabe. Betrachten Sie es als Routing-Hinweis, nicht als Beweis.
</Callout>

## Der Staat ist das Rückgrat der Schleife

Der Kontext ist das, was das Modell **in dieser Runde** sieht. State ist der kompakte Datensatz, der es ermöglicht, die nächste Runde fortzusetzen, ohne die Arbeit zu wiederholen oder zu vergessen.

<Checklist title="Nützlicher Schleifenstatus" items={[
  { text: "Das aktuelle Ziel und die Akzeptanzkriterien" },
  { text: "Bereits unternommene Aktionen und deren beobachtbare Ergebnisse" },
  { text: "Produzierte Artefakte mit Versionen oder stabilen Referenzen" },
  { text: "Gutachterurteile und ungelöste Probleme" },
  { text: "Budget verbraucht und Berechtigungen noch verfügbar" },
  { text: "Der Grund für das Fortfahren, Stoppen oder Eskalieren" },
]} />

Speichern Sie prägnante Entscheidungen und Beweise – nicht die private Gedankenkette eines Modells. Ein guter Zustand ist klein, inspizierbar und kann sicher wieder aufgenommen werden.

## Gängige Schleifenformen

### Die Reparaturschleife

„reproduzieren → eine Sache ändern → Prüfungen durchführen → diagnostizieren → wiederholen oder stoppen“.

Ideal für Code, Konfiguration, Datenbereinigung und alle Aufgaben mit ausführbarem Feedback.

### Die Evaluator-Optimizer-Schleife

„generieren → mit einer Rubrik punkten → gezieltes Feedback zurückgeben → überarbeiten“.

Am besten geeignet für Schreiben, Übersetzen, Designkritik und Ergebnisse, deren Qualität durch artikuliertes Feedback verbessert wird.

### Die Forschungsschleife

„Suchen → Quellen prüfen → Lücken oder Konflikte identifizieren → erneut suchen → synthetisieren“.

Am besten, wenn die Vollständigkeit nicht im Voraus bekannt ist. Die Stoppregel sollte die Evidenzabdeckung messen, nicht die Anzahl der Suchergebnisse.

### Der vom Menschen gesteuerte Kreislauf

„Vorbereiten → automatisch überprüfen → vor risikoreichen Maßnahmen pausieren → menschliche Genehmigung oder Weiterleitung“.

Am besten geeignet für Zahlungen, Veröffentlichungen, Löschungen, Zugriffsänderungen, medizinische oder rechtliche Entscheidungen und andere Folgemaßnahmen.

## Eine minimale Implementierung

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

Der Code ist der einfache Teil. Die schwierigen Fragen sind Domänenfragen: Welcher Test beweist, dass der Fehler behoben ist? Welche Quelle ist maßgeblich? Welche Aktion ist reversibel? Wer kann den letzten Schritt genehmigen? Diese Entscheidungen sind die eigentliche Arbeit des Loop-Engineerings.

Eine zuverlässige Schleife hat ein messbares Ziel, begrenzte Aktionen, vertrauenswürdige Beobachtungen, einen expliziten Bewerter, einen kompakten Zustand, sichere Ausgänge und menschliches Urteilsvermögen, wo Konsequenzen dies erfordern. Das Modell befindet sich innerhalb der Schleife; Der Ingenieur bleibt für die Schleife verantwortlich.
</Callout>
