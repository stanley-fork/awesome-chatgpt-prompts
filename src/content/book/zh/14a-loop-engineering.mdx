上下文工程决定**模型可以看到什么**。循环工程决定**系统下一步做什么**。

循环不再需要一个人重复阅读答案并编写下一个提示，而是将后续工作转变为设计好的系统。它给人工智能一个目标，让它行动，观察真实的反馈，评估结果，然后适应或停止。

<Callout type="info" title="简短的定义">
循环工程是围绕人工智能系统设计反馈循环的实践：目标、行动、观察、评估、状态、护栏和停止规则，使工作朝着可验证的结果发展。
</Callout>

## 从好的提示到好的循环

提示可以产生强有力的第一次尝试。当无法提前知道步数、环境可能发生变化或者必须对照证据检查第一次尝试时，循环非常有用。

这个想法建立在早期的代理模式之上。 [ReAct 论文](https://arxiv.org/abs/2210.03629) 展示了将行动与外部环境观察相结合的价值。 Anthropic 的[评估器优化器模式](https://www.anthropic.com/engineering/building-effective-agents) 添加了一个独特的反馈步骤，而 OpenAI 的[代理实用指南](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) 将代理运行描述为由退出条件控制的循环。较新的术语**循环工程**将注意力集中在故意设计整个循环上。

## 首先编写循环合约

在选择模型或框架之前，先写一份小合同。如果这些字段是模糊的，则循环会将模糊性转化为重复成本。

```yaml
goal: "什么样的可观察状态应该成为真实的？"
inputs: "什么开始循环？"
state: "在迭代之间持续存在哪些事实和尝试？"
actions: "系统可以使用哪些工具以及具有哪些权限？"
observations: "这些动作会返回哪些外部信号？"
evaluator: "哪些评估标准或确定性检查会判断结果？"
success: "有什么证据证明目标已经完成？"
failure: "哪些情况需要康复或人工帮助？"
budget: "最大回合数、时间、代币、金钱或副作用"
```

<Callout type="warning" title="没有出口的循环是一种失败模式">
始终定义三个出口：当证据满足目标时**成功**，当恢复不再有用时**失败**，当循环达到允许的成本或风险边界时**预算耗尽**。
</Callout>

## 将制造者与检查者分开

对于低风险工作，可以起草并自我审查一种模型。对于重要的工作，请使用具有不同工作和有限权限的检查员。

<InfoGrid columns={2} items={[
  { label: "创客", description: "提出变更建议，调用行动工具，并解释应收集哪些证据。", color: "amber" },
  { label: "检查者", description: "接收目标、工件和证据；应用该标题；无法默默地重写其评分的作业。", color: "green" },
]} />

检查员不必是另一个人工智能。首选可用的最具确定性的评估器：

1. **精确检查** — 类型、模式、约束、权限、哈希
2. **可执行检查** — 测试、linter、模拟、链接验证
3. **Rubric 检查** — 使用命名标准的独立模型或人员
4. **结果检查** - 随着时间的推移真实的用户行为或生产指标

<Callout type="tip" title="信心不是证据">
由制作工件的同一模型生成的置信度分数仍然是模型输出。将其视为路由提示，而不是证据。
</Callout>

## 状态是循环的支柱

上下文是模型**本轮**看到的内容。状态是一个紧凑的记录，可以让下一个回合继续进行，而不会重复或忘记工作。

<Checklist title="有用的循环状态" items={[
  { text: "当前目标和验收标准" },
  { text: "已尝试的行动及其可观察的结果" },
  { text: "生成的工件，带有版本或稳定参考" },
  { text: "评估者的结论和未解决的问题" },
  { text: "已消耗预算且权限仍然可用" },
  { text: "继续、停止或升级的原因" },
]} />

存储简洁的决策和证据——而不是模型的私人思想链。好的状态是小、可检查且可以安全恢复的。

## 常见的循环形状

### 修复循环

`复制→改变一件事→运行检查→诊断→重复或停止`

最适合代码、配置、数据清理以及任何具有可执行反馈的任务。

### 评估器-优化器循环

`生成 → 用评分标准评分 → 返回有针对性的反馈 → 修改`

最适合写作、翻译、设计评论和通过明确反馈提高质量的输出。

### 研究循环

`搜索→检查来源→识别差距或冲突→再次搜索→综合`

当事先不知道完整性时最好。停止规则应该衡量证据覆盖范围，而不是搜索结果的数量。

### 人控循环

`准备→自动验证→在高风险行动之前暂停→人工批准或重定向`

最适合付款、发布、删除、访问更改、医疗或法律决定以及其他后续行动。

## 最低限度的实施

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

代码是最简单的部分。难题是领域问题：哪个测试证明错误已经消失？哪个来源是权威的？哪个动作是可逆的？谁可以批准最​​后一步？这些决定是循环工程的实际工作。
