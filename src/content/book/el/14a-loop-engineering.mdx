Η μηχανική περιβάλλοντος αποφασίζει **τι μπορεί να δει το μοντέλο**. Η μηχανική βρόχου αποφασίζει **τι θα κάνει το σύστημα στη συνέχεια**.

Αντί ένα άτομο να διαβάζει επανειλημμένα μια απάντηση και να γράφει την επόμενη προτροπή, ένας βρόχος μετατρέπει αυτή την εργασία παρακολούθησης σε ένα σχεδιασμένο σύστημα. Δίνει ένα στόχο σε μια τεχνητή νοημοσύνη, την αφήνει να δράσει, παρατηρεί την πραγματική ανατροφοδότηση, αξιολογεί το αποτέλεσμα και είτε προσαρμόζεται είτε σταματά.

<Callout type="info" title="Ο σύντομος ορισμός">
Η μηχανική βρόχου είναι η πρακτική του σχεδιασμού του κύκλου ανάδρασης γύρω από ένα σύστημα AI: στόχος, ενέργειες, παρατηρήσεις, αξιολόγηση, κατάσταση, προστατευτικά κιγκλιδώματα και κανόνες διακοπής που οδηγούν την εργασία προς ένα επαληθεύσιμο αποτέλεσμα.
</Callout>

## Από μια καλή προτροπή σε έναν καλό βρόχο

Μια προτροπή μπορεί να δημιουργήσει μια δυνατή πρώτη προσπάθεια. Ένας βρόχος είναι χρήσιμος όταν ο αριθμός των βημάτων δεν είναι γνωστός εκ των προτέρων, το περιβάλλον μπορεί να αλλάξει ή η πρώτη προσπάθεια πρέπει να ελεγχθεί με στοιχεία.

Ο νεότερος όρος **μηχανική βρόχου** εστιάζει την προσοχή στη σκόπιμη σχεδίαση ολόκληρου του κύκλου.

## Γράψτε πρώτα το συμβόλαιο βρόχου

Πριν επιλέξετε ένα μοντέλο ή πλαίσιο, γράψτε ένα μικρό συμβόλαιο. Εάν αυτά τα πεδία είναι ασαφή, ο βρόχος θα μετατρέψει την ασάφεια σε επαναλαμβανόμενο κόστος.

```yaml
goal: "Ποια παρατηρήσιμη κατάσταση πρέπει να γίνει αληθινή;"
inputs: "Τι ξεκινά ο βρόχος;"
state: "Ποια γεγονότα και απόπειρες επιμένουν μεταξύ των επαναλήψεων;"
actions: "Ποια εργαλεία μπορεί να χρησιμοποιήσει το σύστημα και με ποιες άδειες;"
observations: "Ποια εξωτερικά σήματα επιστρέφουν από αυτές τις ενέργειες;"
evaluator: "Ποια ρουμπρίκα ή ντετερμινιστικοί έλεγχοι κρίνουν το αποτέλεσμα;"
success: "Ποια στοιχεία αποδεικνύουν ότι ο στόχος είναι ολοκληρωμένος;"
failure: "Ποιες καταστάσεις απαιτούν ανάρρωση ή ανθρώπινη βοήθεια;"
budget: "Μέγιστες στροφές, χρόνος, μάρκες, χρήματα ή παρενέργειες"
```

<Callout type="warning" title="Ένας βρόχος χωρίς έξοδο είναι μια λειτουργία αποτυχίας">
Να ορίζετε πάντα τρεις εξόδους: **επιτυχία** όταν τα στοιχεία ικανοποιούν τον στόχο, **αποτυχία** όταν η ανάκτηση δεν είναι πλέον χρήσιμη και **εξαντλείται ο προϋπολογισμός** όταν ο βρόχος φτάσει στο επιτρεπόμενο όριο κόστους ή κινδύνου.
</Callout>

Το πούλι δεν χρειάζεται να είναι άλλο ένα AI. Προτιμήστε τον πιο ντετερμινιστικό διαθέσιμο αξιολογητή:

1. **Ακριβείς έλεγχοι** — τύποι, σχήματα, περιορισμοί, δικαιώματα, κατακερματισμοί
2. **Εκτελέσιμοι έλεγχοι** — δοκιμές, linters, προσομοιώσεις, επικύρωση συνδέσμων
3. **Έλεγχοι ρουμπρίκων** — ένα ανεξάρτητο μοντέλο ή άνθρωπος που χρησιμοποιεί επώνυμα κριτήρια
4. **Έλεγχοι αποτελεσμάτων** — πραγματική συμπεριφορά χρήστη ή μετρήσεις παραγωγής με την πάροδο του χρόνου

<Callout type="tip" title="Η εμπιστοσύνη δεν είναι απόδειξη">
Ένας δείκτης εμπιστοσύνης που δημιουργείται από το ίδιο μοντέλο που δημιούργησε το τεχνούργημα εξακολουθεί να αποτελεί έξοδο μοντέλου. Αντιμετωπίστε το ως υπόδειξη δρομολόγησης, όχι ως απόδειξη.
</Callout>

## Το State Is the Spine of the Loop

Το πλαίσιο είναι αυτό που βλέπει το μοντέλο **αυτή τη στροφή**. Κατάσταση είναι το συμπαγές ρεκόρ που αφήνει την επόμενη στροφή να συνεχίσει χωρίς να επαναλάβει ή να ξεχάσει την εργασία.

## Κοινά σχήματα βρόχου

### Ο βρόχος επισκευής

«αναπαραγωγή → αλλαγή ενός πράγματος → εκτέλεση ελέγχων → διάγνωση → επανάληψη ή διακοπή».

Το καλύτερο για κώδικα, διαμόρφωση, εκκαθάριση δεδομένων και οποιαδήποτε εργασία με εκτελέσιμη ανάδραση.

### Ο βρόχος αξιολόγησης-βελτιστοποίησης

`δημιουργία → βαθμολογία με ρουμπρίκα → επιστροφή στοχευμένων σχολίων → αναθεώρηση`

Ιδανικό για γραφή, μετάφραση, κριτική σχεδιασμού και αποτελέσματα των οποίων η ποιότητα βελτιώνεται μέσω της αρθρωτής ανατροφοδότησης.

### Ο βρόχος της έρευνας

`αναζήτηση → επιθεώρηση πηγών → εντοπισμός κενών ή διενέξεων → αναζήτηση ξανά → σύνθεση`

Το καλύτερο όταν η πληρότητα δεν είναι γνωστή εκ των προτέρων. Ο κανόνας διακοπής θα πρέπει να μετρά την κάλυψη αποδεικτικών στοιχείων και όχι τον αριθμό των αποτελεσμάτων αναζήτησης.

### The Human-Gated Loop

"προετοιμασία → αυτόματη επαλήθευση → παύση πριν από ενέργεια υψηλού κινδύνου → εγκρίνει ή ανακατευθύνει τον άνθρωπο"

Το καλύτερο για πληρωμές, δημοσίευση, διαγραφή, αλλαγές πρόσβασης, ιατρικές ή νομικές αποφάσεις και άλλες επακόλουθες ενέργειες.

## Μια ελάχιστη υλοποίηση

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

Ο κώδικας είναι το εύκολο κομμάτι. Οι δύσκολες ερωτήσεις είναι ερωτήσεις τομέα: Ποια δοκιμή αποδεικνύει ότι το σφάλμα έχει εξαφανιστεί; Ποια πηγή είναι έγκυρη; Ποια δράση είναι αναστρέψιμη; Ποιος μπορεί να εγκρίνει το τελικό βήμα; Αυτές οι αποφάσεις είναι το πραγματικό έργο της μηχανικής βρόχου.
