컨텍스트 엔지니어링은 **모델이 볼 수 있는 것**을 결정합니다. 루프 엔지니어링은 **시스템이 다음에 수행할 작업**을 결정합니다.

사람이 반복적으로 답변을 읽고 다음 프롬프트를 작성하는 대신 루프는 후속 작업을 설계된 시스템으로 전환합니다. AI에게 목표를 부여하고, 행동하게 하고, 실제 피드백을 관찰하고, 결과를 평가하고, 적응하거나 중지합니다.

<Callout type="info" title="짧은 정의">
루프 엔지니어링은 검증 가능한 결과를 향해 작업을 이동시키는 목표, 조치, 관찰, 평가, 상태, 가드레일 및 중지 규칙 등 AI 시스템을 중심으로 피드백 주기를 설계하는 관행입니다.
</Callout>

## 좋은 프롬프트에서 좋은 루프로

프롬프트는 강력한 첫 번째 시도를 만들어낼 수 있습니다. 루프는 단계 수를 미리 알 수 없거나 환경이 변할 수 있거나 첫 번째 시도를 증거와 대조해야 할 때 유용합니다.

이 아이디어는 이전 에이전트 패턴을 기반으로 합니다. [ReAct 논문](https://arxiv.org/abs/2210.03629)은 외부 환경의 관찰과 인터리빙 작업의 가치를 보여주었습니다. Anthropic의 [평가자-최적화자 패턴](https://www.anthropic.com/engineering/building-effective-agents)은 고유한 피드백 단계를 추가하는 반면, OpenAI의 [에이전트에 대한 실무 가이드](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)는 종료 조건에 따라 제어되는 루프로 에이전트 실행을 설명합니다. 새로운 용어인 **루프 엔지니어링**은 전체 사이클을 의도적으로 설계하는 데 주의를 집중합니다.

## 루프 계약을 먼저 작성하세요

모델이나 프레임워크를 선택하기 전에 작은 계약서를 작성하세요. 이러한 필드가 모호한 경우 루프는 모호함을 반복 비용으로 바꿉니다.

```yaml
goal: "어떤 관찰 가능한 상태가 참이 되어야 할까요?"
inputs: "루프를 시작하는 것은 무엇입니까?"
state: "반복 간에 어떤 사실과 시도가 지속됩니까?"
actions: "시스템은 어떤 도구를 사용할 수 있으며 어떤 권한을 가지고 있나요?"
observations: "그러한 행동으로부터 어떤 외부 신호가 되돌아오나요?"
evaluator: "어떤 루브릭 또는 결정론적 검사로 결과를 판단합니까?"
success: "목표가 완전하다는 것을 증명하는 증거는 무엇입니까?"
failure: "회복이나 사람의 도움이 필요한 조건은 무엇입니까?"
budget: "최대 턴, 시간, 토큰, 돈 또는 부작용"
```

<Callout type="warning" title="종료가 없는 루프는 실패 모드입니다.">
항상 세 가지 출구를 정의하십시오. 증거가 목표를 충족하는 경우 **성공**, 복구가 더 이상 유용하지 않은 경우 **실패**, 루프가 허용된 비용 또는 위험 경계에 도달하면 **예산 소진**입니다.
</Callout>

## 일반적인 루프 모양

### 수리 루프

`재현 → 한 가지 변경 → 점검 실행 → 진단 → 반복 또는 중지`

코드, 구성, 데이터 정리 및 실행 가능한 피드백이 있는 모든 작업에 가장 적합합니다.

### 평가자-최적화자 루프

`생성 → 루브릭으로 점수 매기기 → 타겟 피드백 반환 → 수정`

명확한 피드백을 통해 품질이 향상되는 글쓰기, 번역, 디자인 비평 및 결과물에 가장 적합합니다.

### 연구 루프

`검색 → 소스 검사 → 차이 또는 충돌 식별 → 다시 검색 → 합성`

완전성을 미리 알 수 없는 경우에 가장 좋습니다. 중지 규칙은 검색 결과 수가 아니라 증거 범위를 측정해야 합니다.

### 인간 게이트 루프

`준비 → 자동 확인 → 고위험 작업 전 일시 중지 → 사람이 승인하거나 리디렉션`

결제, 게시, 삭제, 액세스 변경, 의료 또는 법적 결정, 기타 결과적인 조치에 가장 적합합니다.

## 최소한의 구현

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

코드는 쉬운 부분입니다. 어려운 질문은 도메인 질문입니다. 버그가 사라졌음을 증명하는 테스트는 무엇입니까? 어떤 출처가 신뢰할 수 있나요? 어떤 작업을 되돌릴 수 있나요? 최종 단계는 누가 승인할 수 있나요? 이러한 결정은 루프 엔지니어링의 실제 작업입니다.
