L'ingénierie du contexte décide **de ce que le modèle peut voir**. L'ingénierie de boucle décide **de ce que le système fera ensuite**.

Au lieu qu'une personne lise une réponse à plusieurs reprises et rédige l'invite suivante, une boucle transforme ce travail de suivi en un système conçu. Il donne un objectif à une IA, la laisse agir, observe les retours réels, évalue le résultat et s'adapte ou s'arrête.

<Callout type="info" title="La courte définition">
L'ingénierie de boucle est la pratique consistant à concevoir le cycle de rétroaction autour d'un système d'IA : l'objectif, les actions, les observations, l'évaluation, l'état, les garde-fous et les règles d'arrêt qui font avancer le travail vers un résultat vérifiable.
</Callout>

## D'une bonne invite à une bonne boucle

Une invite peut produire une bonne première tentative. Une boucle est utile lorsque le nombre d'étapes ne peut pas être connu à l'avance, que l'environnement peut changer ou que la première tentative doit être vérifiée par rapport à des preuves.

Cette idée s’appuie sur des modèles d’agents antérieurs. Le [article ReAct] (https://arxiv.org/abs/2210.03629) a montré l'intérêt d'entrelacer des actions avec des observations d'un environnement externe. Le [modèle d'évaluateur-optimiseur] (https://www.anthropic.com/engineering/building-effective-agents) d'Anthropic ajoute une étape de rétroaction distincte, tandis que le [guide pratique des agents] d'OpenAI (https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) décrit l'exécution des agents comme des boucles régies par des conditions de sortie. Le terme plus récent **ingénierie de boucle** concentre l'attention sur la conception délibérée de l'ensemble de ce cycle.

## Rédigez d'abord le contrat de boucle

Avant de choisir un modèle ou un framework, rédigez un petit contrat. Si ces champs sont vagues, la boucle transformera l’ambiguïté en coût répété.

```yaml
goal: "Quel état observable devrait devenir vrai ?"
inputs: "Qu'est-ce qui démarre la boucle ?"
state: "Quels faits et tentatives persistent entre les itérations ?"
actions: "Quels outils le système peut-il utiliser et avec quelles autorisations ?"
observations: "Quels signaux externes renvoient de ces actions ?"
evaluator: "Quelles rubriques ou contrôles déterministes jugent le résultat ?"
success: "Quelles preuves prouvent que l’objectif est atteint ?"
failure: "Quelles conditions nécessitent une guérison ou une aide humaine ?"
budget: "Nombre maximum de tours, de temps, de jetons, d'argent ou d'effets secondaires"
```

<Callout type="warning" title="Une boucle sans sortie est un mode d'échec">
Définissez toujours trois sorties : **succès** lorsque les preuves satisfont l'objectif, **échec** lorsque la récupération n'est plus utile et **budget épuisé** lorsque la boucle atteint sa limite de coût ou de risque autorisée.
</Callout>

## Les observations doivent venir du monde

Le modèle disant « cela semble correct » ne constitue pas une preuve solide. Une observation utile est produite par quelque chose en dehors de la réponse jugée.

## Séparez le créateur du vérificateur

Pour les travaux à faible risque, un modèle peut rédiger et s’auto-évaluer. Pour les travaux importants, utilisez un vérificateur avec un travail différent et une autorité limitée.

<InfoGrid columns={2} items={[
  { label: "Fabricant", description: "Propose le changement, appelle des outils d’action et explique quelles preuves doivent être collectées.", color: "amber" },
  { label: "Vérificateur", description: "Reçoit l'objectif, l'artefact et les preuves ; applique la rubrique ; ne peut pas réécrire silencieusement le travail qu’il note.", color: "green" },
]} />

Le vérificateur ne doit pas nécessairement être une autre IA. Préférez l’évaluateur le plus déterministe disponible :

1. **Contrôles exacts** — types, schémas, contraintes, autorisations, hachages
2. **Contrôles exécutables** — tests, linters, simulations, validation de lien
3. **Vérifications de rubriques** — un modèle indépendant ou un humain utilisant des critères nommés
4. **Contrôles des résultats** : comportement réel des utilisateurs ou mesures de production au fil du temps

<Callout type="tip" title="La confiance n'est pas une preuve">
Un score de confiance généré par le même modèle qui a créé l'artefact est toujours un résultat du modèle. Traitez-le comme un indice de routage, pas comme une preuve.
</Callout>

## L’État est l’épine dorsale de la boucle

Le contexte est ce que le modèle voit **ce tour**. State est le disque compact qui permet de continuer le prochain tour sans répéter ni oublier le travail.

<Checklist title="État de boucle utile" items={[
  { text: "L'objectif actuel et les critères d'acceptation" },
  { text: "Actions déjà tentées et leurs résultats observables" },
  { text: "Artefacts produits, avec versions ou références stables" },
  { text: "Verdicts des évaluateurs et problèmes non résolus" },
  { text: "Budget consommé et autorisations toujours disponibles" },
  { text: "La raison de la poursuite, de l'arrêt ou de l'escalade" },
]} />

Stockez des décisions et des preuves concises, et non la chaîne de pensée privée d'un modèle. Un bon état est petit, inspectable et peut reprendre en toute sécurité.

## Formes de boucles courantes

### La boucle de réparation

`reproduire → changer une chose → exécuter des vérifications → diagnostiquer → répéter ou arrêter`

Idéal pour le code, la configuration, le nettoyage des données et toute tâche avec des commentaires exécutables.

### La boucle évaluateur-optimiseur

`générer → noter avec une rubrique → renvoyer des commentaires ciblés → réviser`

Idéal pour l'écriture, la traduction, la critique de conception et les résultats dont la qualité s'améliore grâce à des commentaires articulés.

### La boucle de la recherche

`rechercher → inspecter les sources → identifier les lacunes ou les conflits → rechercher à nouveau → synthétiser`

Idéal lorsque l’exhaustivité n’est pas connue à l’avance. La règle d’arrêt doit mesurer la couverture des preuves, et non le nombre de résultats de recherche.

### La boucle fermée par l'homme

`préparer → vérifier automatiquement → faire une pause avant une action à haut risque → l'homme approuve ou redirige”

Idéal pour les paiements, la publication, la suppression, les modifications d'accès, les décisions médicales ou juridiques et autres actions consécutives.

## Une mise en œuvre minimale

```typescript
let state = initialize(goal, acceptanceCriteria, budget);

while (state.budget.remaining > 0) {
  const context = assembleRelevantContext(state);
  const proposedAction = await maker.chooseAction(context);
  const observation = await tools.executeWithinPolicy(proposedAction);
  const verdict = await evaluator.check({ goal, observation, state });

  state = recordIteration(state, { proposedAction, observation, verdict });

  if (verdict.status === "passed") return complete(state);
  if (verdict.status === "unsafe" || verdict.status === "blocked") {
    return escalateToHuman(state);
  }

  state = adaptPlan(state, verdict.feedback);
}

return stopWithBudgetReport(state);
```

Le code est la partie la plus facile. Les questions difficiles sont des questions de domaine : quel test prouve que le bug a disparu ? Quelle source fait autorité ? Quelle action est réversible ? Qui peut approuver la dernière étape ? Ces décisions constituent le véritable travail de l’ingénierie des boucles.

## Résumé

<Callout type="tip" title="Créez des commentaires, pas seulement une invite">
Une boucle fiable a un objectif mesurable, des actions limitées, des observations fiables, un évaluateur explicite, un état compact, des sorties sûres et un jugement humain lorsque les conséquences l'exigent. Le modèle est à l’intérieur de la boucle ; l'ingénieur reste responsable de la boucle.
</Callout>
